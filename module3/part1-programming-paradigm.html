Part 1 — Programming Paradigm

Part 1. Programming Paradigm

A Programming Paradigm is the "style" of programming. There are many different paradigms and sometimes they overlap. Some of the main ones are:

The Imperative Paradigm are sequences of commands that are executed one by one as in: first do this, then do that, etc. The programs we have written so far follow this paradigm.

The Functional Paradigm uses mathematical functions as a core principle of building a program. Functions can then be called to evaluate an expression and use the resulting value for further calculations using functions. Functions can also be nested, i.e., a function that calls another function etc. We have used functions earlier, so imagine an almost exclusive use of functions throughout your program.

The Object Oriented Programming (OOP) Paradigm organize the properties and functionalities of something into logical units by creating "object". Still too abstract to understand? Let's think of it in a real-world scenario.

For almost all real-world objects, they are all "defined" by two characteristics: they all have state, and the all have behavior. State describes what the object is, behavior defines what the object can do (functionality).

Take your laptop as an example, the state of a laptop can be described by size, brand, color, cpu type, and many other attributes. The behavior of a laptop can be defined by perform computational task, run program, connect to internet, play music, and many other functionalities. In OOP paradigm, we follow this idea and build "object" ourselves.

All the states are referred to as "attributes", all the behaviors (or functionalities) are refereed to as "methods". We use the class method to define the blueprint for object. Probably even more confused? Don't worry, let define a blueprint and build some objects together.
